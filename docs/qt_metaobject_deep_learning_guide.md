## 1. 先回答你最关心的问题：项目有没有用 Qt 元对象能力
有，而且使用面很广。

关键证据（来自当前仓库）：
- src/ui/tray_icon.py 里 SystemTrayManager(QObject) 定义了多个 Signal。
- src/ui/entity_window.py 里定义了多个状态迁移信号并配合 QStateMachine。
- src/core/audio_manager.py 里 @Slot 装饰器和跨线程 worker 连接是典型元对象通信。
- src/core/director.py 里大量 connect(...) 把模块拼成事件网络。
- src/main.py 里 _PushToTalkBridge(QObject) 通过 Signal 与主流程通信。

你可以把“Qt 元对象能力”先理解为：
- 对象有运行时类型信息。
- 对象可以声明信号。
- 对象可以声明槽函数。
- 信号和槽可在同线程或跨线程安全通信。
- 运行时连接关系可组织成系统级事件流。

目前这个项目没有明显使用的部分：
- 没看到显式 QMetaObject API 的直接调用。
- 没看到 Q_PROPERTY/Property 这种属性系统的核心使用。

这不影响“已经使用元对象能力”的结论。
因为 QObject + Signal + Slot + connect 本身就是元对象系统最常见、最核心的落地方式。

## 2. 概念地图：把名词变成一张可运行的图
你可以把当前程序看成四层。

- 第1层：输入层：鼠标键盘空闲检测、音频输出检测、语音唤醒、摄像头视线。
- 第2层：编排层：Director + StateMachine 做决策、定时、退避和状态切换。
- 第3层：表现层：EntityWindow、Tray、动画粒子、弹窗。
- 第4层：执行层：AudioManager/Worker 线程、QTimer 调度、Qt 事件循环。

在这四层里，元对象系统承担“总线”的角色：
- 输入层 emit 信号。
- 编排层接收并判定。
- 表现层响应并渲染。
- 执行层通过线程和事件循环保证时序与线程安全。

一个最小链路（抽象示例）：
- IdleMonitor 发现用户离开。
- IdleMonitor 发出 user_idle_confirmed。
- Director 的槽函数 on_user_idle 被调用。
- Director 决定状态切换到 ENGAGED 或保持 HIDDEN。
- EntityWindow 根据状态信号切动画。

你在 Go 里的类比：
- Signal 类似“发布事件”。
- connect 类似“注册订阅者”。
- emit 类似“向订阅者广播消息”。
- Qt 会帮你处理线程边界的派发策略。

## 4. Qt 元对象系统基础：你必须真正吃透的 20 个点
- 元对象要点 1：QObject 是元对象系统的基类。
- 元对象要点 2：Signal 在类定义期声明，在运行时可发射。
- 元对象要点 3：Slot 是可连接回调，可带参数签名。
- 元对象要点 4：connect 建立信号到槽的连接关系。
- 元对象要点 5：emit 触发信号，参数将被传递给槽。
- 元对象要点 6：同线程连接通常直接调用槽（Direct）。
- 元对象要点 7：跨线程连接通常进入目标线程事件队列（Queued）。
- 元对象要点 8：QObject 的线程归属由 moveToThread 控制。
- 元对象要点 9：QThread 本身不是工作逻辑容器，worker 模式更常见。
- 元对象要点 10：QTimer 依赖事件循环，线程里要有 event loop。
- 元对象要点 11：信号参数类型越明确，调试越容易。
- 元对象要点 12：一个信号可连多个槽，一个槽也可接多个信号。
- 元对象要点 13：disconnect 可解除连接，避免悬挂回调。
- 元对象要点 14：deleteLater 应在对象所属线程安全销毁 QObject。
- 元对象要点 15：父子对象关系决定生命周期。
- 元对象要点 16：Qt 的事件循环是调度中心。
- 元对象要点 17：元对象能力不是“花哨特性”，而是架构主骨架。
- 元对象要点 18：槽函数要保持短小，重任务放 worker 线程。
- 元对象要点 19：信号命名要语义化，例如 audio_playing_started。
- 元对象要点 20：排错优先看“信号发没发、连没连、线程对不对”。

## 5. 从你的项目看 Signal/Slot 的真实价值
这里不空讲理论，直接映射到仓库里的真实业务。

- 业务案例 1：托盘菜单点击后发信号，由 Director 或主程序处理。
- 业务案例 2：空闲监测线程持续上报 idle 时间，由 Director 决策。
- 业务案例 3：音频输出监测通过轮询 worker 回传状态。
- 业务案例 4：语音唤醒监听触发后上报识别文本和错误。
- 业务案例 5：动画窗口内部状态机通过内部信号迁移状态。
- 业务案例 6：主程序把多个模块连接成全局控制流。

为什么这比“直接函数互调”更好：
- 解耦：发出者不用知道接收者细节。
- 可扩展：后续新增监听者，不需要改发出者。
- 可测试：可以在测试里替换接收端并观察信号。
- 跨线程：框架帮你做线程边界调度。

## 6. 线程、定时器、事件循环：桌面应用的执行模型
你提到 CPU 执行顺序，这里给你一个工程化模型。

主线程：
- 运行 Qt 事件循环。
- 负责 UI 绘制与大部分对象调度。
- 不能被耗时任务阻塞。

工作线程：
- 放耗时或周期性任务。
- 通过信号把结果发回主线程。
- 不直接操作不属于本线程的 UI 对象。

QTimer：
- 不是“精确定时器”，是“事件循环驱动的调度器”。
- 主线程忙时，定时回调会延迟。
- 定时器只保证“尽快”而非“硬实时”。

你在 Go 的类比：
- 主线程像主 goroutine 的 event loop。
- worker 像独立 goroutine。
- Signal/Slot 像 channel + 回调注册的混合体。

## 7. 与 Go / Remix / PostgreSQL 概念的映射
把迁移学习做透，你会进步最快。

- 映射 1：Go channel -> Qt Signal/Slot。都是消息驱动通信；Qt 更偏对象事件模型
- 映射 2：Go context cancel -> Qt 停止信号/状态位。都需要显式中断长任务
- 映射 3：Remix 组件通信 -> QObject 间信号连接。都要管理数据流方向
- 映射 4：Remix hydration -> Qt 首帧初始化。都要关注初始化顺序和可见状态一致
- 映射 5：PG 事务一致性 -> 状态机原子迁移。都强调状态转换的边界条件
- 映射 6：PG 事件通知 -> Qt 事件与信号。都适合解耦模块
- 映射 7：最终一致性 -> 异步信号延迟可见。都要接受“短暂不一致”窗口
- 映射 8：同步调用 -> DirectConnection。调用方等待被调执行完成
- 映射 9：异步队列 -> QueuedConnection。消息入队后稍后处理
- 映射 10：读模型投影 -> UI 状态映射。业务状态要投影成可展示状态

你要刻意练习的一点：
- 每看到一个 Qt 结构，都问自己：这在 Go/Remix/PG 里对应什么范式。
- 这样可以把“知识孤岛”变成“可迁移能力”。

## 10. 排错剧本：当你怀疑“信号没生效”时
- 排错步骤 1：先确认对象是否仍然存活。
- 排错步骤 2：确认 connect 是否真的执行过。
- 排错步骤 3：确认 emit 是否被调用。
- 排错步骤 4：确认信号参数签名与槽匹配。
- 排错步骤 5：确认槽是否抛异常被吞掉。
- 排错步骤 6：确认线程归属是否符合预期。
- 排错步骤 7：确认事件循环是否在运行。
- 排错步骤 8：确认是否被后续逻辑覆盖状态。
- 排错步骤 9：确认是否有多重连接导致重复触发。
- 排错步骤 10：确认测试环境是否正确模拟 Qt 应用对象。

建议日志格式：
- [module] [signal] emit payload=...
- [module] [slot] enter state=...
- [thread] current=... target=...
- [state] from=... to=... reason=...

- 决策逻辑：它如何根据输入更新状态。
- 输出事件：它发出了哪些信号或副作用。
- 线程模型：这些代码运行在哪个线程。
- 失败路径：异常、错误、超时如何处理。
- 可测试性：哪些部分可以单测，哪些需要集成测试。
- 变更风险：改动一处可能影响哪些链路。
- 观测点：需要补什么日志才更好排查。

## 附录 A：面向初学者的最小代码骨架（伪代码）
```python
class Producer(QObject):
    updated = Signal(int)
    def tick(self):
        self.updated.emit(1)

class Consumer(QObject):
    @Slot(int)
    def on_updated(self, value: int):
        print(value)

p = Producer()
c = Consumer()
p.updated.connect(c.on_updated)
p.tick()
```

说明：这段骨架就是元对象通信的最小闭环。
